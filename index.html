Project management tools are powerful.
Until you use them.
Project management tools are powerful. OpenProject, Jira, HubSpot — they can do almost anything. But "can do" and "actually get things done" are separated by a simple gap: the manual effort.

Setting up a project properly — creating work streams, work packages, backlogs, sprints, assigning members, configuring versions, linking dependencies, writing descriptions, organizing categories — takes hours.

Not because it's hard. Because it's tedious. And tedious work gets skipped or postponed.

The result? Projects start with a Gantt chart that looks great on Monday and is outdated by Wednesday.

I tried, by chance to fix that.

Enter OpenClaw
I have learned about OpenClaw in February ... after reading the documentation I decided to pass this technology — I thought it was again a useless agent.

My colleague, Jef Minsart from XFlowdata told me that he was participating to a meetup.com event organized by Toon Vanagt OpenClaw User Group Belgium.

Jef "forced" me indirectly to study the technology.

In one sentence, OpenClaw is a Personal AI Assistant (PAT) running like a Linux Server : 24/7.

The user interacts with AI Agents through any Messaging platform like WhatsApp, Telegram, Discord, Signal, Slack, Mattermost.

The difference with Chat Bots like ChatGPT, Gemini or Claude is that the server is alive and can interact with other systems even if you are not "connected".

For example, every morning I receive "the list of tasks updated in last 72h" in my feed.

The OpenProject Skill: What It Does
The openproject-by-altf1be skill gives OpenClaw full operational access to an OpenProject instance via the API v3. OpenProject is a powerful Opensource Program/Project management system. We use it for our internal projects at XFlowdata.
Core
Work packages — create, read, update, delete, filter
Projects — create, configure, manage members and roles
Members & roles — add users, assign roles
Versions/milestones — create sprints, set date ranges
Categories — organize work by domain
Relations — dependency chains (precedes, blocks, follows)
Extended
Comments & attachments — add context to any work package
News — publish project announcements
Notifications, time entries, watchers, wiki pages
Custom fields, groups, documents, reminders
All of this is accessible through natural language in a Discord channel.

Why This Matters
01
Projects start properly
The #1 killer of project management quality is incomplete setup. When creating a work package takes 2 minutes of clicking, people create vague one-liners with no description, no assignee, no version, no dependencies. When creating a work package takes 0 effort — because you describe it in natural language and the agent handles the rest — every work package gets a proper description, acceptance criteria, and metadata from day one.
02
Consistency at scale
I manage multiple projects across OpenProject, Jira, HubSpot, and GitHub. The OpenClaw skill approach means I interact with all of them the same way: through conversation. The agent handles the API differences. Need to mirror a user's memberships across 14 projects? "make Béatrice member of the same projects as Jeanne" — Done. Seven new memberships created, all with matching roles. In one sentence.
03
Documentation happens automatically
Every work package I create through OpenClaw and my Skills gets: A structured description, Steps and acceptance criteria, Links to relevant resources (SharePoint, GitLab, docs), Proper categorization and versioning. Not because I'm disciplined. Because the agent does it by default. I master the theory and learned from the best. Now the machine enforces what I always meant to do.
04
Discord becomes the control plane
My #bot-openproject Discord channel is now an operations console. I can: Create and manage projects, Query work package status, Generate reports and export them as files, Add members, set roles, configure versions — All without opening a browser. This matters for speed, but it matters even more for continuity. Every command and its result is visible in the chat history. It's a natural audit trail.

The Jira Skill:
Same Philosophy, Different Beast
OpenProject is one side of my project management stack. But I also use Atlassian Jira. So I built a second skill: atlassian-jira-by-altf1be. Same architecture. Same design principles. Different API.
$ openclaw "move PROJ-42 to Done"
● Transitioning PROJ-42... ✓ Done # No browser. No board. No drag-and-drop.
$ "find all open bugs assigned to me in PROJ, ordered by priority"
→ project = PROJ AND status != Done AND type = Bug AND assignee = currentUser() ORDER BY priority DESC
What It Covers
The Jira skill gives OpenClaw full CRUD access to Jira Cloud via the REST API v3:
• Issues — create, read, update, delete, list, and search with JQL
• Comments — add, update, delete, list on any issue
• Attachments — upload, list, delete
• Workflow transitions — list available transitions and move issues through states

That last one matters more than it sounds. Workflow transitions in Jira are where tickets go to die — stuck in "In Review" because nobody clicked the button.
The Same Patterns Work
Everything I described for OpenProject applies to Jira:
• Creating issues with full descriptions, priorities, and assignees — from a sentence
• Bulk operations across projects
• Attaching files from the chat
• Moving issues through workflows without opening a browser
• Commenting on issues as part of a conversation
• Add those work items into a new sprint and start the sprint !

The mental model is identical: describe what you want, the agent handles the API.
Security: Same Standards
Both skills share the same security posture:
• Auth — email + API token (Basic auth, base64 encoded)
• No secrets to stdout — tokens never appear in logs or output
• Destructive operations require --confirm — no accidental deletes
• Rate limiting with exponential backoff — respects Atlassian's API limits
• Lazy config validation — credentials are only checked when you actually run a command
• Path traversal prevention — file uploads are sandboxed
Why Two Skills Instead of One? OpenProject and Jira are fundamentally different systems. Different APIs, different data models, different workflow engines. The agent layer (OpenClaw) provides the unified interface.

What I Learned
01
AI never ever replace project managers. It removes the friction.
The decisions are still mine: what to build, who to assign, how to prioritize. But the execution of those decisions — the clicking, the typing, the formatting, the linking — that's gone.
02
Natural language is the best API client.
I don't need to remember Command lines flags or API endpoints like we did or do when we use the command prompts. I say what I want, and the agent figures out the how. Even the LLMs make mistakes !!! When something fails (wrong project, missing member), it tells me why and I fix it.
03
Quality scales when effort drops to zero.
When adding a description costs nothing, every task gets one. When setting dependencies is free, every chain gets linked. The quality of project setup went up dramatically — not because I got better at it, but because the cost of doing it right became negligible.
